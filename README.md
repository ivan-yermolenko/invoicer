# Monefy - Invoice Management System

Monefy — це веб-додаток для створення, перегляду та управління фінансовими інвойсами.

## 🚀 Як запустити проєкт
Для розгортання всього стеку локально вам знадобляться лише встановлені Docker та Docker Compose.
1. Клонуйте репозиторій.
2. Виконайте команду, яка автоматично збере контейнери, запустить базу даних, встановить залежності, запустить міграції та наповнить базу тестовими даними:
   - **Для Linux / macOS** (використовуючи Make):
     ```bash
     make setup
     ```
   - **Для Windows**:
     Запустіть файл `setup.bat` подвійним кліком, або виконайте у терміналі:
     ```cmd
     setup.bat
     ```

3. Відкрийте додаток у браузері:
   - **Frontend (Nuxt 4)**: [http://localhost:3000](http://localhost:3000)
   - **Backend API (Laravel)**: [http://localhost:8088/api](http://localhost:8088/api)

Інші корисні команди:
- `make up` - запустити контейнери (в фоні)
- `make down` - зупинити контейнери
- `make logs` - переглядати логи контейнерів
- `make test` - запустити автоматичні тести бекенду

## 📝 Відповіді на запитання (Архітектура та прийняті рішення)

### 1. Як ти структурував frontend і backend?

**Backend (Laravel 13 + PostgreSQL)**:
- Класична RESTful API архітектура.
- **FormRequests** (напр. `StoreInvoiceRequest`, `UpdateInvoiceRequest`) повністю ізолюють бізнес-валідацію від контролерів.
- **Resources** (`InvoiceResource`) стандартизують JSON-відповіді.
- Використані **Enums** (`InvoiceStatus`, `InvoiceCurrency`) для жорсткої типізації на рівні додатку (PHP). У базі даних ці поля зберігаються як звичайні `string` / `varchar`. Це сучасний підхід у Laravel, який дозволяє уникнути проблем із жорстким блокуванням схеми БД при додаванні нових значень (як це буває з нативними ENUM у базах даних).

**Frontend (Nuxt 4 + Vue 3.5 + TailwindCSS 4)**:
- Використана архітектура **App Shell (SSR + CSR)**. Сервер швидко генерує "скелет" сторінки та початкові дані (`useApiFetch`), а далі сторінка працює як SPA.
- Перевикористовувані компоненти (наприклад, `components/Invoice/Form.vue` для створення та редагування інвойсів).
- Запроваджено єдиний `useInvoicesApi` composable, який абстрагує всі мережеві запити до бекенду.
- Валідація форм реалізована через зв'язку **Vee-Validate + Zod**, що дублює бекенд-валідацію для миттєвого UX.

### 2. Які компроміси ти зробив і чому?

- **Автентифікація**: Свідомо не додавав Laravel Sanctum / JWT, щоб максимально сфокусуватися на ідеальній реалізації основного флоу інвойсів (CRUD).
- **Спрощена архітектура БД**: Замість створення окремих таблиць для `Suppliers` або `Items` (позицій в інвойсі), ці дані поки зберігаються в таблиці `invoices`. Це пришвидшує MVP, але в майбутньому знадобиться нормалізація БД.
- **Дизайн**: Для швидкої розробки використані чисті утилітні класи TailwindCSS без розгортання великих UI-бібліотек (типу HeadlessUI), що дозволило мати повний контроль над дизайном.

### 3. Що б ти покращив у production-версії?
- **Авторизація та Мультитенантність**: Налаштував би авторизацію (через Sanctum) і можливість створювати інвойси від імені різних компаній (tenants).
- **Оптимізація бази даних**: Додав би **БД індекси (`$table->index(...)`)** на найпопулярніші поля для фільтрації та пошуку (наприклад, `status`, `supplier_name`, `issue_date`), щоб гарантувати високу швидкість запитів при збільшенні об'єму даних.
- **Фільтрація та Пошук**: Додав би можливість фільтрації (за статусом, датою, постачальником) та пошук інвойсів на бекенді, а також підтримку infinite scroll як альтернативу пагінації (базова серверну пагінація вже реалізована).
- **E2E тестування**: Покрив би критичні флоу за допомогою Playwright.
- **Генерація PDF та фонові черги**: Додав би Laravel Horizon та Redis для генерації PDF-версій інвойсів та відправки нотифікацій по email.

### 4. Які UX edge cases ти врахував?
- **Заморозка форми**: При зміні статусу на `approved` або `rejected` форма автоматично блокується (поля стають `readonly`/`disabled`, а кнопка збереження зникає), щоб унеможливити зміну закритих фінансових документів.
- **Миттєвий розрахунок**: Поле "До сплати" (`gross_amount`) динамічно перераховується в реальному часі на основі введених `Net` і `VAT` сум.
- **Попередження незворотних дій**: При спробі змінити статус інвойсу (дія, після якої він блокується), браузер показує `confirm`-вікно, захищаючи від випадкового кліку.
- **Миттєва валідація**: Zod-схема повністю віддзеркалює бекенд, тому невірні дані блокуються миттєво з підсвіткою червоним, навіть не відправляючи зайвий запит на сервер.
- **Скелетони**: Замість стандартних спінерів використовується плавна анімація скелетонів під час очікування даних з API.
- **Hydration Mismatch у Nuxt**: Для запитів даних чітко вказано кастомний ключ (кешу), який синхронізує серверний Payload із клієнтським завантаженням, щоб запобігти смиканню UI на старті.
